# AGENTS.md

- Keep inference profiling opt-in via `profile_inference`; default inference behavior should not emit profile artifacts.
- Do not commit generated beatmaps, audio files, model weights, or `*.profile.json` outputs.
- For expensive profiling, use a GPU host and the `configs/inference/profile_salvalai.yaml` style of reproducible Hydra config rather than running local Mac inference.
- The current inference optimization objective is same-performance, faster inference on RTX 2080/2080 Ti: do not claim wins from changed precision, sampling policy, output policy, model quality, windowing/overlap, or generated-token behavior unless the run is explicitly labeled non-equivalent.
- Exact fixed-seed generated-token identity remains the default acceptance standard for same-calculation speedups. If Mapperatorinator's default generation nondeterminism makes exact token identity too strict, a candidate may be reported only as `documented-drift`/non-equivalent: prove the config and semantics are otherwise unchanged, characterize the drift with repeated baseline-vs-baseline and baseline-vs-candidate runs, keep generated-token counts/output policy aligned, and require an explicitly documented large operational speedup before asking to keep it. Do not silently mix documented-drift results into same-calculation tables or retained baselines.
- Measure before and after every optimization. Preserve fixed-seed generated token IDs for smoke equivalence when practical, separate true model time from torch-profiler overhead, and ask why each change worked before keeping it.
- Always check that performance has not degraded at all before promoting a change. For full-song candidates, compare the current retained baseline against the candidate for main generation, timing generation, total profiled stage time, generated-token counts, fixed-seed token equivalence, per-window main-generation timings, and any touched non-server/server path. If any meaningful same-config performance metric regresses, label it as a scoped regression and ask before accepting it.
- Use `utils/summarize_inference_profile.py --compare BASELINE CANDIDATE --strict --json-output REPORT.json` as the default profile comparison gate. It exits nonzero on same-calculation metadata mismatch, missing generated-token IDs, token mismatch, aggregate regression, generated-token/record-count mismatch, total stage-wall regression, or per-window regression for the selected label. For full-song promotion, run it at least for `main_generation`; inspect timing-generation and total-stage regressions before acceptance.
- Start scouting with bounded smoke profiles such as `configs/inference/profile_salvalai_smoke.yaml`; promote only promising exact-calculation changes to full-song SALVALAI runs.
- Keep optimizations that improve RTX 2080 full-song main-generation throughput by about 10% or more. Keep 5-10% wins only when they are simple and well-contained; remove 1-3% complexity by default.
- Commit and push clean checkpoints for accepted wins, and remove marginal experiments regularly so the branch remains easy to roll back.
- Write durable experiment notes under `notes/`, especially accepted wins, failed/marginal ideas, why a change worked, why it was reverted, and the DCC job/profile paths that support the decision.
- Do not reintroduce rejected exact-calculation experiments such as final-position-only logits projection, static-cache SDPA prefix trimming, persistent static-mask buffer mutation, dynamic/default-cache generation, forced `CompileConfig(dynamic=False)`, forced `CompileConfig(dynamic=True)`, forced `CompileConfig(fullgraph=True)`, forced `CompileConfig(mode="max-autotune")`, copy-compatible custom `_sample` monkeypatching, preallocated `_sample` loop replacement, old pre-graph stateful monotonic time masking, or `torch.inference_mode` generation wrapping unless new profiler evidence explains why the old negative or non-equivalent result no longer applies.
- Prioritize optimizations that improve the inference engine in general: generation loop overhead, cache behavior, batching/windowing, model-call structure, memory movement, and profiling clarity. Treat backend-specific work as secondary unless profiling shows it is the actual bottleneck.
- Improvements that help future batch inference or multiple-song throughput matter too. Measure and document them separately from cold single-song throughput, reporting first-song cold cost, warmed subsequent-song cost, aggregate multi-song throughput, token equivalence per song, and any batching/windowing behavior changes.
- Use `utils/profile_inference_suite.py` for same-process warm-repeat or serial multi-song scouting. For `serial_multi_song`, pass an explicit `--song-list` YAML/JSON/text file and keep token equivalence per song. Treat warmed results as `warm_repeat`/multi-song evidence only; they do not replace the cold single-song baseline unless separately validated through the normal cold full-song gates.
- Suite manifests must make cold tax visible. Preserve `main_first_record`, `main_remaining_records`, warmed/all aggregate segment metrics, timing-context metrics, per-song metrics, and runtime cache/env metadata so active-prefix, batch, and serving results cannot hide first-window compile/graph/setup costs, timing regressions, or uneven song-level regressions. Use `utils/summarize_inference_profile.py --compare-suite BASE_MANIFEST CANDIDATE_MANIFEST --suite-scope warmed_runs --strict` for warmed-suite gates, and use `--suite-scope all_runs` only when the claim is all-run aggregate performance. Add `--gate-cold-run0` only when the claim includes cold run0 non-regression.
- Treat 5+ songs as the common operational batch/multi-song workload, not an edge case. For realistic batch/multi-song scouting, prefer suites of at least 5 songs when feasible. If `batch_mode` and `normal_mode` diverge materially, keep separate mode-specific baselines, gates, and result tables rather than mixing them into cold single-song or warmed-repeat claims.
- Use explicit `attn_implementation=sdpa` as the current inference/profiling baseline on DCC. Do not frame SDPA as a permanent commitment; revisit FA2 or other backends only if broader inference-engine profiling indicates a likely win.
- Use `inference_generation_compile=true` and `inference_active_prefix_decode_loop=false` for the current retained cold single-song full-song inference profiling baseline unless an experiment explicitly measures compile-disabled, active-prefix, batch, or legacy behavior. Account for one-time compile/specialization costs separately from post-warmup decode throughput, but never accept a cold single-song change without a full-song non-regression check against the retained compile-only baseline.
- The active long-range inference research target is `200 tok/s` full-song main-generation throughput on RTX 2080/2080 Ti. Treat it as a deeper runtime/kernel project, not permission to relax same-calculation rules.
- Planned future architecture context from OliBomby: current Mapperatorinator is one transformer that autoregressively generates beatmaps directly from spectrogram audio; the possible future direction is autoregressive transformers plus latent diffusion in a Mapperatorinator-esque encoder-decoder with compression/diffusion. Prioritize inference lessons that carry to the encoder-decoder/autoregressive component; diffusion/compression may change the bottleneck mix.
- For the 200 tok/s phase, start quick iteration with a middle-15-second SALVALAI smoke slice such as `configs/inference/profile_salvalai_smoke15.yaml`; promote to 30s or full-song runs only after fixed-seed token equivalence and a plausibly meaningful speed signal.
- Before custom decoder-step, graph, backend, or kernel work claims speed, run `utils/verify_one_token_decode.py --sequence-index 9` on the 15s smoke config. It compares a direct static-cache `q_len=1` decode step against captured raw logits from HF cached `generate()` and is a logits gate only; full generated-token equivalence is still required. Use `--include-no-cache-diagnostics` only when debugging the ABI; it adds slower no-cache forwards that are not part of the pass condition.
- For custom/direct decode loop changes, run `utils/verify_direct_decode_loop.py` before 15s smoke promotion. It compares normal HF `generate()` against the candidate loop via `custom_generate`, checks generated-token identity, raw-logit equality, and final RNG-state equality over multiple sampled steps, and should be used as a fast correctness gate rather than a throughput claim.
- For manual CUDA graph direct-loop work, use `utils/verify_direct_decode_loop.py --candidate-cuda-graph-forward` as a verifier-only gate before any smoke or speed claim. Job `49165810` showed active-prefix bucket512 graph replay can advance through 8 sampled decode steps with compile disabled/enabled while preserving generated tokens, RNG, logits allclose, and top-k order; job `49165980` extended the graph gate to 64 tokens; jobs `49166191` and `49166213` validated bucket512->bucket1024 graph recapture over 512 generated tokens for compile disabled/enabled. This does not prove inference throughput because the verifier still pays setup, input preparation, tensor copies, and short-run overhead.
- Use `osuT5.osuT5.inference.direct_decode` as the reusable one-token direct-step ABI for custom runtime/CUDA-graph experiments. Job `49139917` validated `seq9` raw-logit equivalence through this helper with compile disabled and enabled; do not fork a separate direct-step implementation unless the helper is proven insufficient.
