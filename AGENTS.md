# AGENTS.md

- Keep inference profiling opt-in via `profile_inference`; default inference behavior should not emit profile artifacts.
- Do not commit generated beatmaps, audio files, model weights, or `*.profile.json` outputs.
- For expensive profiling, use a GPU host and the `configs/inference/profile_salvalai.yaml` style of reproducible Hydra config rather than running local Mac inference.
- The current inference optimization objective is same-performance, faster inference on RTX 2080/2080 Ti: do not claim wins from changed precision, sampling policy, output policy, model quality, windowing/overlap, or generated-token behavior unless the run is explicitly labeled non-equivalent.
- Measure before and after every optimization. Preserve fixed-seed generated token IDs for smoke equivalence when practical, separate true model time from torch-profiler overhead, and ask why each change worked before keeping it.
- Always check that performance has not degraded at all before promoting a change. For full-song candidates, compare the current retained baseline against the candidate for main generation, timing generation, total profiled stage time, generated-token counts, fixed-seed token equivalence, per-window main-generation timings, and any touched non-server/server path. If any meaningful same-config performance metric regresses, label it as a scoped regression and ask before accepting it.
- Start scouting with bounded smoke profiles such as `configs/inference/profile_salvalai_smoke.yaml`; promote only promising exact-calculation changes to full-song SALVALAI runs.
- Keep optimizations that improve RTX 2080 full-song main-generation throughput by about 10% or more. Keep 5-10% wins only when they are simple and well-contained; remove 1-3% complexity by default.
- Commit and push clean checkpoints for accepted wins, and remove marginal experiments regularly so the branch remains easy to roll back.
- Write durable experiment notes under `notes/`, especially accepted wins, failed/marginal ideas, why a change worked, why it was reverted, and the DCC job/profile paths that support the decision.
- Do not reintroduce rejected exact-calculation experiments such as stateful monotonic time masking, final-position-only logits projection, static-cache SDPA prefix trimming, persistent static-mask buffer mutation, dynamic/default-cache generation, forced `CompileConfig(dynamic=False)`, forced `CompileConfig(dynamic=True)`, forced `CompileConfig(fullgraph=True)`, forced `CompileConfig(mode="max-autotune")`, copy-compatible custom `_sample` monkeypatching, preallocated `_sample` loop replacement, or `torch.inference_mode` generation wrapping unless new profiler evidence explains why the old negative or non-equivalent result no longer applies.
- Prioritize optimizations that improve the inference engine in general: generation loop overhead, cache behavior, batching/windowing, model-call structure, memory movement, and profiling clarity. Treat backend-specific work as secondary unless profiling shows it is the actual bottleneck.
- Improvements that help future batch inference or multiple-song throughput matter too. Measure and document them separately from cold single-song throughput, reporting first-song cold cost, warmed subsequent-song cost, aggregate multi-song throughput, token equivalence per song, and any batching/windowing behavior changes.
- Use `utils/profile_inference_suite.py` for same-process warm-repeat or serial multi-song scouting. For `serial_multi_song`, pass an explicit `--song-list` YAML/JSON/text file and keep token equivalence per song. Treat warmed results as `warm_repeat`/multi-song evidence only; they do not replace the cold single-song baseline unless separately validated through the normal cold full-song gates.
- Treat 5+ songs as the common operational batch/multi-song workload, not an edge case. For realistic batch/multi-song scouting, prefer suites of at least 5 songs when feasible. If `batch_mode` and `normal_mode` diverge materially, keep separate mode-specific baselines, gates, and result tables rather than mixing them into cold single-song or warmed-repeat claims.
- Use explicit `attn_implementation=sdpa` as the current inference/profiling baseline on DCC. Do not frame SDPA as a permanent commitment; revisit FA2 or other backends only if broader inference-engine profiling indicates a likely win.
- Use `inference_generation_compile=true` and `inference_active_prefix_decode_loop=false` for the current retained cold single-song full-song inference profiling baseline unless an experiment explicitly measures compile-disabled, active-prefix, batch, or legacy behavior. Account for one-time compile/specialization costs separately from post-warmup decode throughput, but never accept a cold single-song change without a full-song non-regression check against the retained compile-only baseline.
- The active long-range inference research target is `200 tok/s` full-song main-generation throughput on RTX 2080/2080 Ti. Treat it as a deeper runtime/kernel project, not permission to relax same-calculation rules.
- Planned future architecture context from OliBomby: current Mapperatorinator is one transformer that autoregressively generates beatmaps directly from spectrogram audio; the possible future direction is autoregressive transformers plus latent diffusion in a Mapperatorinator-esque encoder-decoder with compression/diffusion. Prioritize inference lessons that carry to the encoder-decoder/autoregressive component; diffusion/compression may change the bottleneck mix.
- For the 200 tok/s phase, start quick iteration with a middle-15-second SALVALAI smoke slice such as `configs/inference/profile_salvalai_smoke15.yaml`; promote to 30s or full-song runs only after fixed-seed token equivalence and a plausibly meaningful speed signal.
- Before custom decoder-step, graph, backend, or kernel work claims speed, run `utils/verify_one_token_decode.py` on the 15s smoke config. It compares a direct static-cache `q_len=1` decode step against captured raw logits from HF cached `generate()` and is a logits gate only; full generated-token equivalence is still required.
- Use `osuT5.osuT5.inference.direct_decode` as the reusable one-token direct-step ABI for custom runtime/CUDA-graph experiments. Job `49139917` validated `seq9` raw-logit equivalence through this helper with compile disabled and enabled; do not fork a separate direct-step implementation unless the helper is proven insufficient.
- Use `profile_generation_detail_ranges=true` only for torch-profiler/Nsight diagnosis. It adds NVTX/`record_function` ranges inside VarWhisper decoder self-attention, cross-attention, MLP, final norm, and output projection, and should not be used for untraced throughput claims.
- Use `profile_nvtx_generation_ranges=true` for Nsight Systems diagnosis when top-level `generation.<label>.seqN` ranges are needed without enabling torch.profiler. This is diagnostic-only instrumentation and must not be used to justify throughput by traced wall time.
- Use `profile_sdpa_backend=flash|efficient|math|cudnn` only for SDPA dispatch audits. Record PyTorch warnings, actual kernel names, token equivalence, and speed; do not replace the SDPA baseline unless full-song token-equivalent results justify it.
- Do not promote forced `profile_sdpa_backend=math` from 15s smoke alone. Full-song job `49139420` was token-equivalent but `89.684 tok/s`, slower than the retained compile-only `92.465 tok/s` baseline; treat forced math SDPA as rejected for the current main-generation objective unless new traces explain why that result is stale.
- Do not promote TorchDynamo `recompile_limit`/`cache_size_limit` tuning as a main-generation optimization. Smoke job `49139564` raised both to `32`, passed token equivalence, and changed main-generation speed only `50.850 -> 50.867 tok/s`.
- Custom runtime/decode-loop work must prove token identity in stages before any speed claim graduates: compile-disabled 15s smoke equivalence, compile-enabled 15s smoke equivalence, then full-song equivalence. Preserve HF generation semantics including top-p/top-k, temperature processors, EOS behavior, RNG consumption, static cache behavior, and generated-token accounting.
- Manual CUDA graph replay of the current fixed direct one-token forward is useful but not enough for `200 tok/s` by itself. Job `49139948` measured exact logits and about `8.09ms` per graph replay, roughly `124 tok/s` before real generation-loop work. Treat graphing as a path toward the `120 tok/s` milestone unless paired with attention/cache-layout wins.
- For a plausible `200 tok/s` path, focus on exact active-prefix self-attention/cache layout. Jobs `49139966` and `49139981` showed q_len=1 SDPA `kv2560` with static mask is about `0.374ms` per call while `kv512` is about `0.069ms`; the full static self-attention length is the big lever, not the 4D mask alone.
- Active-prefix self-attention must remain decode-only in any candidate path. Job `49140082` showed one-token decode-only active-prefix passes logits gates with compile disabled and enabled and reduces fixed graph replay to about `3.79ms`, but active-prefix during prefill fails hard (`max_abs≈15.4`, top-k mismatch). Leave prefill on the normal static-cache path unless new evidence explains and fixes that mismatch.
- Bucketed active-prefix decode is currently the preferred graph-shape strategy. Job `49140217` showed decode buckets `128`, `256`, and `512` all passed one-token logits gates with compile disabled/enabled and replayed in about `3.76-4.42ms`; use bucketed shapes before per-token exact-prefix graph captures.
- Bucketed active-prefix decode remains default-off and did not replace the retained cold single-song baseline. Job `49150185` bucket `256` looked like a large exact win (`92.998 -> 121.926 tok/s`), but follow-up jobs showed strong order/warm-state sensitivity: job `49151748` measured bucket `256` cold at `88.699 tok/s` and bucket `512` warm at `108.313 tok/s`; job `49152465` measured bucket `512` cold-first at `97.800 tok/s` with token equivalence PASS. Job `49154643` later showed bucket `512` as a strong exact full-song warm-repeat candidate (`92.207 -> 128.585 tok/s` warmed aggregate, `+39.5%`, paired token equivalence PASS). Job `49155778` repeated the test active-first with isolated TorchInductor/CUDA caches and still measured an exact warmed full-song win (`91.982 -> 128.026 tok/s`, `+39.2%`) with no paired main/timing/stage-wall degradation, but it is still only `warm_repeat`/future batch-serving evidence. Treat active-prefix as a strategic warm/batch/custom-runtime candidate, not a cold baseline, until it beats the retained compile-only baseline by the normal graduation rules without timing/total-stage/per-window regressions. Do not apply active-prefix during prefill or scope it beyond batch-1/non-server/simple generation without fresh equivalence and performance non-regression runs.
- Job `49156700` attributed active-prefix cold weakness to first-window graph/capture/specialization cost, not steady-state decode. Full-song cold active512 had `seq0=25.531s` but `seq1+ = 131.003 tok/s`; warmed repeats had `seq0≈3.8s` and `126-128 tok/s` total. Job `49156749` logged heavy TorchDynamo/CUDAGraph churn (`cache_utils.py:update` recompiling by `layer_idx`, `recompile_limit`, thousands of graph-recording lines). Next active-prefix implementation work should target graph stabilization, explicit graph/cache priming with honest setup accounting, or a bufferized/direct decode loop. Do not start kernel work from this evidence unless a fresh post-stabilization trace shows GPU kernels, not graph/runtime churn, dominate.
