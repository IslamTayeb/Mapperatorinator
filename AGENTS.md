# AGENTS.md

- Keep inference profiling opt-in via `profile_inference`; default inference behavior should not emit profile artifacts.
- Do not commit generated beatmaps, audio files, model weights, or `*.profile.json` outputs.
- For expensive profiling, use a GPU host and the `configs/inference/profile_salvalai.yaml` style of reproducible Hydra config rather than running local Mac inference.
- Treat `inference.py` and `osuT5/osuT5/inference/server.py:model_generate()` as the public inference runtime control plane. User-facing optimization flags must be declared in `config.py`/Hydra defaults, validated in `inference.py`, routed through `server.py:model_generate()` where practical, and surfaced in profile metadata. Low-level helpers/native kernels can live elsewhere, but mode selection, guardrails, failure behavior, and throughput/result metadata should not bypass these files without a documented measured reason.
- When adding or touching inference entry points, forward `inference_generation_compile` and other public runtime flags through `load_model_with_server()`/`load_model_loaders()` instead of relying on defaults. A requested flag must either apply to the loaded model path or fail loudly; profile metadata should distinguish requested flags from context-level enabled flags.
- The current inference optimization objective is same-performance, faster inference on RTX 2080/2080 Ti: do not claim wins from changed precision, sampling policy, output policy, model quality, windowing/overlap, or generated-token behavior unless the run is explicitly labeled non-equivalent.
- Exact fixed-seed generated-token identity remains the default acceptance standard for same-calculation speedups. If Mapperatorinator's default generation nondeterminism makes exact token identity too strict, a candidate may be reported only as `documented-drift`/non-equivalent: prove the config and semantics are otherwise unchanged, characterize the drift with repeated baseline-vs-baseline and baseline-vs-candidate runs, keep generated-token counts/output policy aligned, and require an explicitly documented large operational speedup before asking to keep it. Do not silently mix documented-drift results into same-calculation tables or retained baselines.
- Measure before and after every optimization. Preserve fixed-seed generated token IDs for smoke equivalence when practical, separate true model time from torch-profiler overhead, and ask why each change worked before keeping it.
- Re-check current bottlenecks and theoretical ceilings regularly. Before starting or promoting a new optimization family, refresh the best available synchronized model-time split and compute the fantasy ceiling if the target became free; do not keep chasing stale targets that have already fallen below the 5-10% keep threshold.
- Do not treat a large profiler bucket as an optimization target until the current code path proves the cost is actually avoidable. Some decoder computation is mathematically required; separate removable overhead, launch/layout/cache work, redundant control flow, and data movement above hardware floors from necessary model math, then compute the best-case TPS if only the removable part disappeared before writing runtime or kernel code.
- Every production optimization branch must start with a bottleneck proof: current accepted baseline seconds/TPS, measured exclusive or overlap-aware target time, avoidable fraction, hardware/roofline or graph-replay floor when applicable, fantasy-free and fantasy-floor TPS, 5%/10% keep bars, and the exact reason this target is the next bottleneck rather than a mathematically necessary cost or already-rejected marginal path.
- Before writing production/runtime code for a new optimization family, prove it is still on the current bottleneck path with the latest accepted stack. Verifier ceilings and microbenchmarks can justify one bounded experiment, but if the smoke/full-song synchronized model-time gain is marginal, per-window regressions appear, or the improvement is mostly setup/wall-cache behavior rather than steady decoder TPS, stop, revert the runtime code, and document the rejection instead of iterating around the wrong target.
- Always check that performance has not degraded at all before promoting a change. For full-song candidates, compare the current retained baseline against the candidate for main generation, timing context, total profiled stage time, generated-token counts, fixed-seed token equivalence, per-window main-generation timings, and any touched non-server/server path. If any meaningful same-config performance metric regresses, label it as a scoped regression and ask before accepting it.
- Use `utils/summarize_inference_profile.py --compare BASELINE CANDIDATE --strict --json-output REPORT.json` as the default single-label profile comparison gate. It exits nonzero on same-calculation metadata mismatch, missing generated-token IDs, token mismatch, aggregate regression, generated-token/record-count mismatch, total stage-wall regression, or per-window regression for the selected label. For full-song promotion, prefer `--strict-full-song`, which checks both `main_generation` and `timing_context` in one report and requires generated output artifact hash equivalence, so timing-context regressions or changed `.osu` bytes cannot be skipped accidentally.
- Start scouting with bounded smoke profiles such as `configs/inference/profile_salvalai_smoke.yaml`; promote only promising exact-calculation changes to full-song SALVALAI runs.
- Keep optimizations that improve RTX 2080 full-song main-generation throughput by about 10% or more. Keep 5-10% wins only when they are simple and well-contained; remove 1-3% complexity by default.
- Commit and push clean checkpoints for accepted wins, and remove marginal experiments regularly so the branch remains easy to roll back.
- Use `main` as the integration branch for accepted work. Start new work from an up-to-date `main` unless there is a specific reason not to. Do trivial accepted docs/guardrail updates directly on `main` when appropriate; use short-lived experiment branches for risky profiling/runtime/kernel work, then merge or cherry-pick them into `main` only after the normal equivalence, no-regression, documentation, and push gates pass. Delete or clearly abandon rejected experiment branches after their result is documented.
- Write durable experiment notes under `notes/`, especially accepted wins, failed/marginal ideas, why a change worked, why it was reverted, and the DCC job/profile paths that support the decision.
- Do not reintroduce rejected exact-calculation experiments such as final-position-only logits projection, static-cache SDPA prefix trimming, persistent static-mask buffer mutation, dynamic/default-cache generation, forced `CompileConfig(dynamic=False)`, forced `CompileConfig(dynamic=True)`, forced `CompileConfig(fullgraph=True)`, forced `CompileConfig(mode="max-autotune")`, copy-compatible custom `_sample` monkeypatching, preallocated `_sample` loop replacement, old pre-graph stateful monotonic time masking, or `torch.inference_mode` generation wrapping unless new profiler evidence explains why the old negative or non-equivalent result no longer applies.
- Prioritize optimizations that improve the inference engine in general: generation loop overhead, cache behavior, batching/windowing, model-call structure, memory movement, and profiling clarity. Treat backend-specific work as secondary unless profiling shows it is the actual bottleneck.
- Improvements that help future batch inference or multiple-song throughput matter too. Measure and document them separately from cold single-song throughput, reporting first-song cold cost, warmed subsequent-song cost, aggregate multi-song throughput, token equivalence per song, and any batching/windowing behavior changes.
- Keep single-song TPS, same-process warm-repeat, static server/window batching, and any future continuous batching as separate modes with separate baselines, flags, gates, and result tables. Continuous batching must preserve each request's generated-token IDs/output behavior, stopping state, cache state, logits-processor state, and RNG behavior for equivalent claims; otherwise label it `documented-drift`/non-equivalent and do not mix it into single-song TPS claims.
- If continuous batching is explored, use explicit batch-specific flags rather than overloading the single-song DecodeSession/native flags. Candidate flags should distinguish scheduler policy, max active sequences, max wait time, prefill policy, decode-order/RNG policy, and any batch-specific DecodeSession/native kernels. Add profile metadata for result class, throughput claim scope, batching mode, request/window IDs, batch IDs, active batch-size histogram, queue wait, per-request latency, prefill/decode split, token hashes, stop reasons, RNG policy/state hashes, cache slot/reorder events, and graph capture/replay counts.
- Use `utils/profile_inference_suite.py` for same-process warm-repeat or serial multi-song scouting. For `serial_multi_song`, pass an explicit `--song-list` YAML/JSON/text file and keep token equivalence per song. Treat warmed results as `warm_repeat`/multi-song evidence only; they do not replace the cold single-song baseline unless separately validated through the normal cold full-song gates.

Preserve `main_first_record`, `main_remaining_records`, warmed/all aggregate segment metrics, timing-context metrics, per-song metrics, output artifact hashes, and runtime cache/env metadata so active-prefix, batch, and serving results cannot hide first-window compile/graph/setup costs, timing regressions, output-byte drift, or uneven song-level regressions.
